sentret — KQL Detection Rule Analyser

A provider-agnostic, LLM-powered analyser for Microsoft Sentinel / Azure Log Analytics detection rules: review existing KQL detections, generate new ones from intent, validate them against a live workspace with bounded, audited queries, and produce a Sentinel-ready rule package. Runs on Anthropic (Claude) or OpenAI — pick the backend with a flag.

Overview

sentret takes a single JSON request file describing either an existing KQL detection rule to analyse (mode: "analyse_existing_rule") or a detection you want created from intent (mode: "create_new_detection"). An LLM agent — Claude or OpenAI, your choice — then works through the full detection-engineering workflow — static KQL review, schema and data-availability checks, live validation queries, performance review with query statistics, threshold tuning, and MITRE ATT&CK mapping — and produces:

  • a verdict (Deploy as-is / Deploy after changes / Test only / Do not deploy) with a 0–100 score,
  • recommended KQL plus alternatives,
  • a complete Sentinel scheduled-rule configuration (severity, frequency, entity mappings, …),
  • a structured JSON report and a human-readable markdown report, saved to disk.
Defensive use only. The tool is for authorised defensive security engineering. Every live query is safety-gated, bounded to an explicit timespan, confined to the requested workspace, redacted by default, and written to an audit log. Nothing is ever deployed to Sentinel without explicit operator approval.

Getting started

Install & build

git clone <repo> && cd sentret
npm install
npm run build          # optional — npx tsx runs straight from source

Choosing a provider

sentret is provider-agnostic. The LLM backend is auto-detected from your environment: an Anthropic key selects Claude (preferred when several are set), then AZURE_OPENAI_ENDPOINT selects Azure AI Foundry, then OPENAI_API_KEY selects OpenAI. Override explicitly with --provider anthropic|openai|azure or SENTRET_PROVIDER. Set the model with --model, SENTRET_MODEL (Anthropic), OPENAI_MODEL, or — on Azure — the deployment name. Point OPENAI_BASE_URL at OpenRouter or a local OpenAI-compatible server (for Azure, use the Azure settings below, not OPENAI_BASE_URL).

Reasoning depth via --effortadaptive thinking + effortreasoning_effort (o-series / GPT-5)reasoning_effort (force with SENTRET_REASONING)
Refusal fallback to a second model✅ (Fable → Opus)— content-filter ends run—

Azure AI Foundry (e.g. GPT-5.5)

Azure routes by deployment name and uses a resource endpoint plus an api-version, so it has its own settings and does not use OPENAI_BASE_URL. The official AzureOpenAI client handles the wire details; auth is an API key if you set one, otherwise Entra ID via DefaultAzureCredential — the same az login identity you already use for Log Analytics. 

export AZURE_OPENAI_ENDPOINT=https://my-foundry.openai.azure.com  # resource URL, not the deployment URL
export AZURE_OPENAI_DEPLOYMENT=gpt-5-5-prod                       # your deployment name
export AZURE_OPENAI_API_VERSION=2025-04-01-preview               # match your deployment
# export AZURE_OPENAI_API_KEY=...    # optional; omit to use Entra ID
az login                                                          # if using Entra ID

npx tsx src/cli.ts --input examples/example_user_request.json     # azure auto-detected from the endpoint
Two settings to get right for GPT-5.5. (1) AZURE_OPENAI_API_VERSION must match what your Foundry deployment serves — the built-in default is a best guess and a wrong value returns a 400. (2) GPT-5.x is a reasoning model, but Azure deployment names are opaque so name-based detection can miss; if reasoning_effort isn't applied, set SENTRET_REASONING=always.
Set your exact model. The default OPENAI_MODEL is a conservative placeholder. On OpenAI set it to the model id your account serves; on Azure set AZURE_OPENAI_DEPLOYMENT to your deployment name.

Authentication

cp .env.example .env   # put ANTHROPIC_API_KEY and/or OPENAI_API_KEY in .env
az login               # or export AZURE_TENANT_ID / AZURE_CLIENT_ID / AZURE_CLIENT_SECRET

A .env file in the working directory is loaded automatically. Never commit it. You only need the key for the provider you'll use; Azure credentials are required only for live workspace queries (skip with --static-only).

Your first run

Copy one of the example requests, set your workspace GUID, and run:

# Analyse an existing rule
npx tsx src/cli.ts --input examples/example_user_request.json

# Create a new detection from intent
npx tsx src/cli.ts --input examples/example_create_new_request.json

# No live queries at all — static analysis only
npx tsx src/cli.ts --input examples/example_user_request.json --static-only

# Compiled CLI (after npm run build)
node dist/cli.js --input request.json

While it runs: the model's working notes and final report stream to stdout; tool invocations (with their audited purpose) and summarized thinking stream to stderr. A run typically issues several validation queries — count checks, schema checks, field-population checks, threshold sensitivity — before submitting its report.

When it finishes you'll see a summary on stderr:

Verdict: Deploy after changes | Score: 71/100 (Needs Tuning)
Saved: reports/analysis_sentret_2026-06-12_ab12cd34.json
Saved: reports/analysis_sentret_2026-06-12_ab12cd34.md

Request file format

Required fields:

FieldDescription
mode"analyse_existing_rule" or "create_new_detection".
detection_intentNatural-language description of the behaviour the rule should detect.
workspace_idLog Analytics workspace GUID. All queries are confined to this workspace.
timespanISO 8601 duration ("PT24H", "P7D") or start/end range ("2026-06-01/2026-06-08").
kqlThe existing query. Required for analyse_existing_rule; omit for create mode.

Optional fields:

FieldDefaultDescription
allow_query_executiontrueWhen false, the run is static-only (same as --static-only).
allow_raw_examplesfalseWhen false, every query result is redacted server-side before the model sees it: emails/UPNs, IPs, GUIDs, URLs, hosts, paths, tokens, secrets, and well-known sensitive columns become deterministic placeholders like <email:1a2b3c4d> that preserve distinctness for aggregation.
sample_mode_allowedtrueWhen false, raw row sampling (take/limit/sample) is blocked entirely — only counts, summaries, and schema checks run.
allow_broad_queriesfalseOperator approval for workspace-wide patterns (search, find, union *, wildcard unions, externaldata). Cross-workspace escapes stay blocked regardless.
data_sensitivity—Low/Medium/High/Restricted. High and Restricted cannot be combined with allow_raw_examples: true — the request is rejected up front.
sentinel_target—{ subscription_id, resource_group, workspace_name }. Required for any Sentinel operation (rule listing or deployment). Verified against the analysed workspace GUID via ARM before first use.
rule_frequency / rule_period—Intended schedule, e.g. "PT1H" / "P1D". Informs frequency/lookback recommendations.
severity_preference—Informational/Low/Medium/High.
noise_tolerance—Low/Medium/High — how many false positives your SOC can absorb.
mitre_tactics / mitre_techniques—Hints for ATT&CK mapping, e.g. ["Credential Access"], ["T1621"].
relevant_data_sources—Candidate tables/connectors, e.g. ["SigninLogs"].
target_platform, environment_context, expected_entities—Free-form context: platform (Microsoft Sentinel, Defender XDR…), environment (cloud-only Entra ID, hybrid…), expected entities (Account, IP, Host…).

Behaviour

Safety controls (enforced in code, not just prompted)

ControlEnforcement
Bounded timespansEvery query requires a valid ISO 8601 duration or date range; rejected otherwise.
Audited purposeEvery query carries a purpose string; executed, blocked, and failed queries are all logged.
Broad-scan blockingKusto control commands (.ingest, .drop, …), workspace-wide search/find, wildcard unions (union *, union App*), and externaldata are blocked unless allow_broad_queries is set. Patterns hidden in string literals or comments neither trigger nor evade the gate.
Workspace confinementQueries run only against the request's workspace_id. Cross-workspace escapes — additional_workspaces, workspace(), cluster(), app(), adx() — are blocked regardless of any approval flag.
Sample capstake/limit/sample over 50 rows is blocked; sample_mode_allowed: false blocks sampling entirely.
Redaction by defaultUnless allow_raw_examples: true, results are deterministically redacted (values, sensitive columns, secret-shaped strings, and error/statistics metadata) before reaching the model.
Result capsRow and character caps on everything returned to the model.
Deployment gatingSee below — three independent gates.

Deploying to Sentinel

Deployment is off by default and triple-gated. To enable it for one run:

  1. Add sentinel_target to the request — the exact subscription, resource group, and workspace name you approve. Before any Sentinel call, the tool resolves this workspace via ARM and verifies its customerId equals the analysed workspace_id; mismatches are blocked.
  2. Pass --allow-deploy on the CLI. Without it the deployment tool is not even offered to the model.
  3. The ARM client itself refuses unconfirmed calls as defence in depth.
npx tsx src/cli.ts --input request.json --allow-deploy
Review before you deploy. The recommended flow is to run without --allow-deploy first, review the generated rule package in the report, and only then re-run with deployment enabled. Deployments (and refused attempts) are written to the audit log with the rule ID and ARM resource ID. MITRE sub-technique IDs are automatically normalised to parent IDs (T1110.003 → T1110) because the Sentinel ARM API rejects dotted IDs.

Rule listing (list_sentinel_rules, used for duplicate detection) needs only sentinel_target, not --allow-deploy.

Refusal fallback Anthropic

Claude Fable 5 runs safety classifiers that occasionally decline benign security-adjacent work. When the primary model refuses, sentret discards any partial output (a visible marker is printed), replays the conversation on the fallback model (default claude-opus-4-8), and continues. The run summary notes which model finished: 

Note: claude-fable-5 declined (category: cyber); completed on fallback model claude-opus-4-8.

Disable with --no-fallback or SENTRET_FALLBACK_MODEL= (empty), in which case a refusal ends the run with exit code 1. On the OpenAI provider there is no fallback: a content-filter stop ends the run (exit 1).

Troubleshooting

SymptomCause / fix
Error: ANTHROPIC_API_KEY is not set (exit 2)Export the key or add it to .env.
Error mentioning Microsoft Entra / az loginNo usable Azure credential. Run az login, or configure a service principal (AZURE_TENANT_ID, AZURE_CLIENT_ID, AZURE_CLIENT_SECRET).
AuthError on queries (401/403)Identity lacks Log Analytics Reader on the workspace, or wrong tenant.
WorkspaceNotFound (404)Check the workspace GUID in the request.
Throttled (429)Log Analytics rate limit; the message includes Retry-After. Re-run later or narrow the queries.
403 on deployment mentioning Sentinel ContributorThe identity needs the Microsoft Sentinel Contributor role on the workspace.
Query blocked: … lines in the transcriptWorking as intended — the safety gate refused a query and told the model why; it normally reformulates and continues. Frequent broad-scan blocks may mean the request needs allow_broad_queries: true.
Sentinel listing blocked: … operator-approved targetThe model tried ARM coordinates that don't match sentinel_target, or the target workspace doesn't resolve to the analysed workspace GUID.
Long silent pausesDeep reasoning phases. Summarized thinking streams to stderr; if you redirected stderr, check that file. Lower --effort for faster, shallower runs.
Sovereign cloud auth failuresSet SENTRET_LOG_ANALYTICS_ENDPOINT / SENTRET_ARM_ENDPOINT and AZURE_AUTHORITY_HOST for your cloud.
Azure: 400 / unsupported api-version or modelSet AZURE_OPENAI_API_VERSION to the version your GPT-5.5 deployment serves, and confirm AZURE_OPENAI_DEPLOYMENT is the deployment name (not the model family).
Azure: GPT-5.5 not reasoning / shallow answersName-based detection can miss opaque deployment names; set SENTRET_REASONING=always to force reasoning_effort.
Azure: 401 / DefaultAzureCredential errorsRun az login (or configure a managed identity / AZURE_* service principal), and grant it the Cognitive Services OpenAI User role on the Foundry resource — or set AZURE_OPENAI_API_KEY to use a key instead.
